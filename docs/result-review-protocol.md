# 结果输出前审查协议

最后更新：2026-07-01

每次实验输出正式结论前，先按下面顺序完成 review。这个文件用于避免把评测设置变化、数据泄漏或重复样本误写成方法提升。

## 固定流程

1. 先写原始结果，只描述指标，不解释贡献。
2. 审查输入、输出、候选集、标签和指标定义。
3. 检查数据泄漏、重复样本、候选池大小变化和字段 ablation。
4. 复查结果变化的原因：模型、数据、候选池、字段还是评测方式。
5. 根据 review 结果迭代脚本或配置。
6. 只把复查后仍成立的结论写入实验登记表和论文路线。

## 必答问题

- 输入是什么？
- 输出是什么？
- 正样本和负样本怎么定义？
- 是否训练？如果训练，loss 是什么？
- 如果结果提高，究竟是模型改变、数据改变、候选集改变，还是指标改变？
- 是否有答案泄漏、同图重复、同文本重复或 split 泄漏？
- baseline 和新方法是否在同一候选池、同一 query 集和同一指标下比较？

## RC1 当前额外规则

- 文本证据主实验不得使用 ScienceQA `solution` 字段作为唯一依据。
- AI2D 图像检索必须报告 sample-level 和 diagram-level 指标。
- wrong-image 评测必须过滤 same-diagram pair。
- hard negative 指标必须说明候选池大小和共同 query 子集。
- 未训练的 TF-IDF / BM25 / CLIP / SigLIP 只能称为 baseline，不得称为方法提升。
